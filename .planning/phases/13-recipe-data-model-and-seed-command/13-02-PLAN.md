    **Write tests first** in `tests/Feature/BlizzardApi/SyncRecipesCommandTest.php` using Pest with RefreshDatabase and Http::fake(). Model the test structure on existing `PriceFetchActionTest.php`. Create Http::fake() responses for all four API levels:
    - Profession index returning 2 professions (one crafting, one gathering to test filtering)
    - Profession detail with multiple skill_tiers (ensure highest-ID is picked)
    - Skill tier recipe list with categories containing 2-3 recipes
    - Recipe detail with full reagents, one with crafted_item present, one with crafted_item missing
    - Profession media endpoint returning icon asset

    Before each test, seed the CatalogItem rows that recipes will reference (reagent items and crafted items). Use CatalogItem::factory() for this.

    **Then implement** `SyncRecipesCommand`:

    1. **Signature:** `blizzard:sync-recipes {--dry-run} {--report-gaps}`

    2. **Description:** `Sync Midnight expansion crafting recipes from the Blizzard API`

    3. **handle(BlizzardTokenService $tokenService)** — main flow:

    a. Get token, region from config, initialize gap stats array and counters (newWatched, existedWatched).

    b. **Level 1 — Profession index:** GET `/data/wow/profession/index`. Filter to crafting professions only: Alchemy (171), Blacksmithing (164), Enchanting (333), Engineering (202), Inscription (773), Jewelcrafting (755), Leatherworking (165), Tailoring (197), Cooking (185). Use a constant array of known crafting profession IDs. Log which professions were found.

    c. **Level 2 — For each crafting profession:**
      - GET `/data/wow/profession/{id}` to get skill_tiers array
      - Select highest-ID skill tier: `collect($data['skill_tiers'])->sortByDesc('id')->first()`
      - Log the selected tier name for gate validation (STATE.md blocker)
      - GET `/data/wow/media/profession/{id}` for icon_url (extract from assets array where key=icon)
      - Upsert Profession: `Profession::updateOrCreate(['blizzard_profession_id' => $id], ['name' => ..., 'icon_url' => ..., 'last_synced_at' => now()])`
      - GET `/data/wow/profession/{id}/skill-tier/{tierId}` to get recipe list
      - Flatten categories[].recipes[] into a single list of recipe IDs/names

    d. **Level 3 — Recipe detail batch (Http::pool()):**
      - Chunk recipe IDs into batches of 20
      - For each batch, Http::pool() GET `/data/wow/recipe/{id}` with token + namespace
      - 1 second pause between batches (usleep(1_000_000))
      - Progress bar matching SyncCatalogCommand format
      - For each successful response:
        - Track gap stats: increment total, check for missing crafted_item and crafted_quantity
        - Handle crafted_item resolution: if `crafted_item` present in response, look up CatalogItem by name using the assignQualityTiers pattern — find matching CatalogItem rows ordered by blizzard_item_id, first = silver (T1), second = gold (T2). Store their IDs as crafted_item_id_silver and crafted_item_id_gold. If no match found, store NULL.
        - Determine is_commodity: if crafted item found in catalog and it has quality_tier, it is likely commodity. If crafted_item name suggests gear (contains "Helm", "Chestguard", etc.) OR the recipe has no crafted_item match, set is_commodity based on best guess. Default to true.
        - Upsert Recipe: `Recipe::updateOrCreate(['blizzard_recipe_id' => $recipeId], [profession_id, name, crafted_item_id_silver, crafted_item_id_gold, crafted_quantity (default 1 if missing), is_commodity, last_synced_at])`. All wrapped in `if (!$dryRun)`.
        - Delete + re-insert reagents: `$recipe->reagents()->delete()` then for each reagent, look up CatalogItem by blizzard_item_id. If not found, create a minimal CatalogItem entry with the reagent name and blizzard_item_id (log warning). Create RecipeReagent row.
        - Auto-watch: For each unique item ID encountered (reagents + crafted items), `WatchedItem::firstOrCreate(['user_id' => 1, 'blizzard_item_id' => $itemId], ['name' => $name, 'profession' => $professionName])`. Track new vs existing counts. Skip if $dryRun.

    e. **After all professions processed:**
      - If --report-gaps: output table using `$this->table()` with headers [Profession, Total, Missing Item, Missing Qty, Coverage %]
      - Print summary: "Synced X recipes (Y professions). Auto-watched Z items (N new, M already existed). Gaps: G recipes missing crafted_item."

    4. **Copy `resolveLocalizedName()` from SyncCatalogCommand** — handles string or {en_US: "..."} name format.

    5. **Handle rate limiting:** If response status is 429, add to retry queue. After batch, retry with 10s pause (mirror SyncCatalogCommand pattern).

    6. **Error handling:** Log individual recipe fetch failures but do not abort the entire sync. Continue to next recipe.

    Follow `declare(strict_types=1)` convention.
