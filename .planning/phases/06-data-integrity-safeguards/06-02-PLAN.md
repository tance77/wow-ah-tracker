    **Test 11: skips snapshot write when Last-Modified absent and hash matches**
    ```php
    it('skips snapshot write via hash fallback when Last-Modified is absent', function (): void {
        fakeBlizzardHttp(); // No Last-Modified header (default)

        // Compute the hash of the fixture response body to pre-seed metadata
        $fixtureBody = file_get_contents(base_path('tests/Fixtures/blizzard_commodities.json'));
        $expectedHash = md5($fixtureBody);

        IngestionMetadata::create([
            'response_hash'   => $expectedHash,
            'last_fetched_at' => now()->subMinutes(15),
        ]);

        $user = User::factory()->create();
        WatchedItem::factory()->create(['user_id' => $user->id, 'blizzard_item_id' => 224025]);

        (new FetchCommodityPricesJob)->handle(app(PriceFetchAction::class), app(PriceAggregateAction::class));

        expect(PriceSnapshot::count())->toBe(0); // Hash gate blocked write
    });
    ```

    IMPORTANT: The hash is computed from the raw body that `$response->body()` returns. When using `Http::response($array, 200)`, Laravel's fake returns the JSON-encoded string of that array. The test must compute the hash from the same string. Use `file_get_contents()` to read the fixture file and compute `md5()` on it. However, there is a subtlety: `Http::response()` receives a decoded array (via `json_decode`), then re-encodes it as JSON for `->body()`. The re-encoded JSON may differ from the original file (whitespace, key ordering). To handle this correctly, compute the hash the same way `PriceFetchAction` will see it:
    ```php
    $fixtureData = json_decode(file_get_contents(base_path('tests/Fixtures/blizzard_commodities.json')), true);
    $expectedHash = md5(json_encode($fixtureData));
    ```
    Use this approach for the hash computation in all hash-related tests.

    ```php
        Http::fake([

                'access_token' => 'test-token',
                'token_type'   => 'bearer',
                'expires_in'   => 86399,
            ], 200),

        ]);
        Cache::forget('blizzard_token');

        expect(PriceSnapshot::count())->toBe(0);

        $meta = IngestionMetadata::first();
        expect($meta->consecutive_failures)->toBe(1);
    });
    ```

    NOTE on retry behavior: `PriceFetchAction` uses `retry(2, 1000, throw: false)`. With `throw: false`, retries happen but no exception is thrown by the HTTP client. Instead, the action checks `$response->successful()` and throws its own `RuntimeException`. The 500 response stub will be retried by the HTTP client, fail again, and then the action throws `RuntimeException`, which the job catches. For tests to avoid waiting for the 1000ms retry delay, this is acceptable since it's only 1-2 seconds. If test speed is a concern, the test can accept the small delay.
